# 下游能力册 —— alarm-api 网关(插件作者视角) (DOWNSTREAM)

> **这份文档给谁看**:在本框架里写"会输出闹钟"的插件的人或 AI。回答一个问题:
> **下游能做什么、你能依赖什么、你千万别做什么。**
> 权威真相源是下游仓库的两份文档:`alarm-api/docs/external-alarms.md`(对外协议 v1)与
> `external-alarms-internal.md`(网关内部机制)。本文是**消费侧摘要 + 本框架的对接现状**,
> 若有出入,以下游文档为准。

---

## 0. 三层分工(铁律:本框架绝不输出 iPhone 概念)

```
calendar-api(本框架)     负责:把业务算成【哪一刻】→ {uid,date,time,reason}
        ↓ HTTP GET ?format=json(闹钟接入协议 v1)
alarm-api(网关 Worker)   负责:时区换算、采纳窗口、拼手机端标签、对账开关建删
        ↓ JSON
iPhone 快捷指令           负责:照单执行(创建/开/关闹钟)
```

标签名(`Gate-ES-*`)、快捷指令语义、设备状态,全是下游领地 —— 插件**永远不需要也不允许**
知道它们;你只吐协议 v1 条目,框架替你包壳。这个分界让三层各自独立演进。

## 1. 本框架已替你实现的协议面(插件零成本获得)

- 出口:任意视图链接 + `?format=json` → `{ v:1, alarms:[{uid,date,time,reason,…}] }`。
- 框架代劳:按日未来过滤(当日保留)、按 date+time 排序、熔断域剔除、字段**原样透传**
  (你多给的协议可选字段如 `tz` 一并输出,不增不删不改)。
- 你的义务(见 DEVGUIDE §5):uid 规范、尊重条目 `remind:false` 与 `alarmOffsets`。

## 2. 下游行为 —— 你可以依赖的事实

| 下游行为 | 对插件的意义 |
|---|---|
| **采纳窗口 ≈ 未来 2 天,多吐无害**(窗口外仅计日志,属正常) | 你按业务窗口尽管全量吐(哪怕一年),不用为下游截窗 |
| **改时间 = 同 uid + 新 time,自动生效**(旧关新建) | 改提前量/假期漂移时,千万保持 uid 不变 —— 这正是 uid 协议"bucket 不含时钟时间"的**下游依据**:网关把最终墙上时间拼进手机标签 `Gate-ES-<code>-<uid>-<HHMM>`,时间由**网关**拼;若你把时间放进 uid,一改时间就成了"新闹钟",旧的变僵尸 |
| **取消 = 不再吐该 uid**(对账自动关) | 无需删除指令;条目 isActive:false / 过期自然消失即可 |
| **时区**:date/time 默认按东八区墙上时间;条目可带 `tz`(IANA 名或 ±HH:MM/Z),网关精确换算(含 DST、可跨天) | 东八区业务直接吐;异时区业务**别自己换算**,带 tz 字段交给网关 |
| **uid 净化**:`[A-Za-z0-9_.-]`,uid≤40、code≤16 | DEVGUIDE 的"纯 ASCII"规范再收紧一档;中文/emoji 进 uid = 被拒(日志计"无uid") |
| **单源 5 秒超时**,慢了本轮跳过(不伤网关主流程,但你这条不下发) | 本框架整条链必须远快于 5s —— DEVGUIDE §8 性能预算的来源 |
| **一条记录 = 一个闹钟** | 界面上怎么合并展示是日历的事;给网关的每条各响各的(信用卡域 merged 模式=业务上合并成一条再吐,合规) |
| 未知字段双向忽略;顶层 `v` 留给协议升级 | 演进空间已留好,别依赖网关回传 |

## 3. 对接登记(每个新增闹钟视图,一次性)

网关侧一条源配置(下游领地,格式见其 internal 文档 §1):

```js
{ name: '还款', type: 'json', code: 'repay', url: 'https://<本Worker>/?cal=card&format=json', enabled: true }
```

- `code` 是手机标签里的项目段,由**下游**为本框架的每条订阅链接分配;换视图=换一条源。
- 公开 URL 进下游 `config.SOURCES`;**带隐私的 URL 走下游 Cloudflare Secret
  `EXTERNAL_ALARMS`**(JSON 数组,格式同上)—— 本框架若做私密视图,记得走这条路登记。
- ICS 途径(`[[ES:uid]]` 标签)下游也支持,但本框架**不用**:JSON 是正门,双输出里闹钟
  流天然独立。唯一可能用它的场景:某插件想让"个别"日历事件顺带变闹钟 —— 那也应优先
  直接产 alarms。
  ⚠️ v6 起这条路**事实上已封死**:域交【事件对象】、不碰 ICS 文本,想塞标签只能塞进
  `summary`/`description`(那是给人看的字段,不是给机器的暗道)。这是好事 ——
  一件事只有一条正门。

## 4. 联调与排错

- **网关侧自检**:在网关调试 URL 加 `?testDate=<某天>`,humanReadable 面板会打一行
  `[外部闹钟] 🌐 还款(repay): 候选N 窗口内新增M (拒/警:无标签a/全天b/无uid c/格式d/窗口外e/时区未识别f)`。
- 常见对号:`无uid` 多 → uid 净化后为空(非 ASCII 主体);`窗口外` 多 → 正常(见 §2 窗口);
  `时区未识别` → tz 值 Intl 不认,已按东八区兜底,去核对写法。
- **本框架侧自检**:同一视图链接 `?format=json&testDate=` 直接看条目;DEVGUIDE §10 自检清单
  第 3、4 条(uid 稳定性/不抖)是网关不抖的充分条件。

## 5. 版本与演进

协议 v1;顶层 `v` 字段是升级钩子。下游承诺 `Gate-ES-` 前缀固定而 code/uid 结构各自可演进。
本框架侧对应约束沉淀在 DEVGUIDE §5(uid 协议)与 ARCHITECTURE §3(透传契约),三处一致,
改任何一处前先对齐另外两处与下游文档。
