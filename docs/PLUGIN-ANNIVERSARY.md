## 0. 为什么是一个域，而不是三个

生日 / 公历纪念日 / 农历纪念日 / 长岁花名册，**共享同一个通讯录、同一套引擎骨架**，
差异被收进两个正交维度（§2）。硬拆成多个域会重复通讯录管理、重复合并逻辑、重复鉴权标记。
**一个 `anniversary` 域，类型是字段不是域。**（与信用卡“三种还款模型一个域”同理。）

域 id：`anniv`（`[a-z][a-z0-9]*`，用于 `?cal=anniv`、颜色派生、uid 前缀）。

## 1. 数据模型：通讯录（两层继承）

**一个人 = 一个档案（联系人）。一个档案挂 N 个“日子”（occasion）。** 像手机通讯录里一个
联系人可以录公历生日、农历生日、多个纪念日。

```
config/anniversary.js（用户领地，private，姓名只在这里，永不进 URL）:

person（档案）:
  id            人的稳定短 id（'mom'）—— 手写、永不改（身份根，见 §5）
  name          显示名（'妈妈'）—— 只进日历标题，不进 URL、不进 uid
  private 默认   整个域 private:true，无需逐人标
  ┌─ 档案级默认（这个人整体的偏好，被 occasion 覆盖）:
  │   leadReminders   日历 VALARM 提前量（如 [7,3,1] 天）
  │   channel         进哪些通道（见 CHANNELS §3.4）
  │   severity        重要度基线
  │   mergeable       是否参与合并（缺省 true）
  └─ occasions[]（挂在这个人下面的日子）:
       oid          —— ⚠️ v6 决策：不手写！由 kind+calendar 自动区分（见 §5）
       kind         事件类型维度（birthday | anniversary | roster …）见 §2
       calendar     历法维度（solar | lunar | 未来 tibetan …）见 §2
       date         精确日期（三种形态，见 §3）
       label        这个日子的名字（'结婚纪念日'；生日可省，用 name 拼）
       （可覆盖档案级的任何默认：单独的 leadReminders/channel/severity/mergeable/政策字段）
```

**两层继承**：档案定默认，occasion 可覆盖。与信用卡 `DEFAULT_CONFIG` 被账户覆盖同一手法。
例：妈妈档案默认 `channel:'calendar'`，但她的“公历生日”这个 occasion 单独设 `channel:'todo'`。

## 2. 两个正交维度（防返工的核心骨架，画死）

**这是整个域最关键的结构决定。事件类型和历法是【两个独立维度】，绝不合并成一个字段。**

```
                       calendar（历法转换器）
                  ┌──────────┬──────────┬─────────────┐
 kind（类型处理器）│  solar   │  lunar   │ tibetan(未来)│
  birthday        │ 公历生日 │ 农历生日 │  （藏历生日）│
  anniversary     │ 结婚纪念 │ 农历纪念 │             │
  roster（花名册）│ 9/1长岁  │ 春节长岁 │             │
```

### 2.1 kind = 事件类型处理器（一个可注册插槽）

每个 `kind` 是一个处理器，实现同一接口：**给定年份 → 算出这天 → 生成标题/正文**。

| kind | 算年龄? | 产出 | 说明 |
|---|---|---|---|
| `birthday` | ✅ | 每年一个事件 | 标题带岁数（政策见 §4） |
| `anniversary` | ❌（或算“第 N 周年”） | 每年一个事件 | 结婚/忌日等，不算“岁” |
| `roster`（长岁花名册） | ✅（列全员） | **一年一条**，列所有人 | 独立类型，不参与合并，见 §6 |

**加新类型 = 加一个处理器，引擎不动。** 未来“纪念日倒计时”“节气提醒”都是新 kind。

### 2.2 calendar = 历法转换器（另一个可注册插槽）

每个 `calendar` 是一个转换器，实现同一接口：
**输入精确出生日 → 析出“每年锚点” → 给定年份算出该年公历日期**。

| calendar | 转换器 | 状态 |
|---|---|---|
| `solar` | 恒等（公历本身） | v1 实现 |
| `lunar` | 中国农历（离线库，见 §7） | v1 实现 |
| `tibetan` / `islamic` / … | 各自转换器 | **预留插槽，不实现**（见 §8） |

**加新历法 = 加一个转换器 + 注册一行，引擎其余不动。** 这是 Ivan “未来精确公历转藏历不返工”
诉求的**具体保证**——不是嘴上兼容，是接口形状挡住返工（同 workdays-core 数据集插件、
registry 域插件模式）。

### 2.3 两维正交的铁律

> **kind 和 calendar 从第一天就是独立字段。** 绝不出现 `type: '公历生日'` 这种把两维焊死的
> 设计——那样加“农历忌日”就得改结构。用测试或代码注释把这条守住（同领地看守精神）。

## 3. 精确日期的三种输入形态（新 AI 最易猜错处）

**“精确”分三种，每种能析出什么不同。** 引擎必须识别输入形态，据此决定能生成哪些历法的锚点。

| 形态 | 输入例 | 能析出的“每年锚点” | 说明 |
|---|---|---|---|
| **A. 精确公历** | `1995-05-05` | 公历 `05-05` **＋** 农历（查表得四月初六） | **一个精确公历日 → 公农历双锚点**。这是 Ivan “维护一个精确日期即可”的核心 |
| **B. 精确农历** | `农历乙亥年四月初六` | 农历 `四月初六` **＋** 公历（查表反推） | 反向同理 |
| **C. 仅公历月日**（无年） | `05-01` | **只有公历** `05-01` | 无年 → 无法双历转换、无法算年龄 |
| **C′. 仅农历月日**（无年） | `农历四月初六` | **只有农历** `四月初六` | 同上，只过农历 |

**关键推论**：
- 形态 A/B **一个精确日期能同时支持公历生日和农历生日**——生成哪个/哪些由 §3.1 的开关定，
  **不需要用户填两个日期**。引擎内部析出双锚点。
- 形态 C/C′ 没有年份 → **不算年龄、上不了长岁花名册**（虚岁需年份，§6）。这是数据能力的
  硬边界，不是 bug，文档写明。
- **闰月出生**（形态 A 落在闰月，或形态 B 本身是闰月）：锚点如何记，见 §4.1（这是农历最深的坑）。

### 3.1 双历开关：一个精确日期，过哪些生日

形态 A/B（有完整信息）的 person，用一个开关决定这个精确生日**生成哪些 occasion**：

```
birthObserve: 'solar' | 'lunar' | 'both'    // 只过公历 / 只过农历 / 都过；缺省实施时定
```

- 这不是“填两个 occasion”，而是**一个精确生日 date + 一个开关** → 引擎析出对应锚点、
  产出对应事件。`both` → 同一个精确生日产出两个事件（公历生日 + 农历生日），各带各的岁数。
- 形态 C/C′ 信息不全，开关被限制（只公历的只能 `solar`，只农历的只能 `lunar`）。

## 4. 政策参数（“怎么过”全进 config，档案默认 + occasion 覆盖）

**所有“怎么过”都是参数，都有缺省。缺省先由本文定，Ivan 开发前过一遍改。**
参数分两级：`config/anniversary.js` 顶部**全局默认** → person 档案级 → occasion 级，**就近覆盖**。

| 政策 | 缺省（可改） | 说明 |
|---|---|---|
| **闰月生日平年怎么过** | **都过**（普通月都提醒） | Ivan 定：默认都提醒；当年有闰月时，日历用字段额外标注。见 §4.1 |
| **农历三十遇小月** | **廿九** | 提前不迟到，同信用卡 2 月溢出哲学 |
| **公历 2·29 平年** | **2·28** | 提前不迟到，与上一致 |
| **岁数显示** | **正文全列**（虚岁 X / 周岁 Y），标题只在整寿显示 | Ivan 定“都放”；但标题保持简单（§9），详情进正文 |
| **整寿（逢十）升级** | 自动升 `severity: high` | 60 大寿等。**按哪个历判整十见 §4.2** |
| **默认通道** | **仅日历**（`channel: 'calendar'`） | Ivan 定；进闹钟/todo 逐人逐occasion 覆盖 |
| **合并** | `mergeable: true`，重要的人设 false | 见 §5 合并 |
| **时区** | 东八区（`tz: 'Asia/Shanghai'`） | 农历按东八定义；`?tz=` 可覆盖，见 §10 |

### 4.1 闰月的“析出”与“过法”（农历最深的坑，单独说）

**两件事，别混：**

**(a) 锚点怎么记**（析出时，一次性）：精确生日落在闰月（如闰四月初六出生），锚点记为
**闰四月初六**（保真出生状态），同时知道其对应普通月 `四月初六`。

**(b) 平年怎么过**（每年生成时）：缺省 **都过**——平年过普通月 `四月初六`；遇到本命闰月年
（下一个闰四月，十几年一遇），**额外**过闰四月初六那天，且**日历用字段标注**
（如标题/正文注明“闰四月”）。

> ⚠️ **给 Ivan 的诚实提醒（已确认，记录在此）**：这个“都过”缺省意味着大部分年份过普通月、
> 闰月年多过一次。另一种世界是“忠实还原闰月、平年跳普通月”——那会让生日在两个相差一个月的
> 日期间**反复横跳**（十几年一次突然晚一个月）。Ivan 选了“都过 + 字段标注”，兼顾稳定与
> 保真。可 config 改。

### 4.2 整寿按哪个历判

**按这个 occasion **自己的历**判整十。** 农历生日按农历年龄判 60，公历生日按公历年龄判 60。
`both` 的人，公历那条和农历那条**可能不同年**触发整寿（年底出生的人公农历年龄差一岁）——
**这是对的，各过各的整寿**。（config 可加“按指定历统一判”的覆盖，缺省各自判。）

## 5. 身份编码（uid 稳定，删一条不牵连其他）

**诉求**：编辑通讯录（删人、删日子、改昵称、修生日）时，**没动的人和日子，uid 不能变**
（否则日历全刷、闹钟/todo 全部重建）。

### 5.1 方案（Ivan 采纳“折中版”）

```
person 手写一个稳定短 id（'mom'）—— 挡同名（两个“张伟”）、挡改昵称漂移
occasion 不手写 id —— 由 kind + calendar 自动区分

uid = anniv-<hash(person.id + '/' + kind + '-' + calendar)>-<年份>
      例：anniv-<uidHash('mom/birthday-lunar')>-2026
```

**约束**（本就是常识，写进校验）：**同一个人，同一 kind + 同一 calendar 只能有一个 occasion**。
妈妈不能有两个“农历生日”——天然唯一，正好当键。

### 5.2 为什么这样

- **不用数组下标**：删中间一个 occasion，其余 uid 不变（下标会移位 → 全变）。
- **不用显示名**：改昵称“妈妈”→“母亲”，uid 不变（名字会漂）。
- **person 手写 id 挡两件事**：同名不同人（两个“张伟”各给 `zhangwei1`/`zhangwei2`）、
  改昵称不影响身份。**每人只写一个 id，occasion 全自动**——负担最小又完全稳定。
- **哈希用 uidHash**（governance.js，crc32 8 位）：uid 短、稳、ASCII，满足下游 ≤40 硬限。
  与信用卡账户哈希同一工具、同一“身份不建在会变属性上”的原则（DEVLOG v6.1）。

### 5.3 跨通道 uid（见 CHANNELS §4）

同一生日进日历+闹钟+todo → 三个 uid，共享 `anniv-<hash>-<年份>` 前缀，肉眼配对。
日历 uid 可含更细日期，闹钟/todo uid 是 bucket（年份粒度，不含时钟时间）。

## 6. 合并（信用卡原样搬）+ 长岁花名册（独立类型）

### 6.1 合并：框架分组设施（域内先行，跨域留位置）

**合并分三层理解：已定的、强倾向的、留给未来的。别混。**

#### (a) 已定：`mergeKey` 域内分组设施

合并是**框架流水线的一个正式环节**（`域产事件 → 治理 → 【合并环节】→ 渲染`），
但**"合并成什么样"永远是域自己写**——框架只做"分组"这个不需要懂业务的机械活。

```
每个事件带可选 mergeKey：
  不填        → 独立，不参与合并（缺省）
  填了        → 同一天 + 同 mergeKey 的事件，交给域的回调揉成一条
框架做：按 (域, 天, mergeKey) 分组                      ← 机械活，框架
域做：  组内多于1个 → 域的 mergeEvents(group) 写合并事件  ← 业务，域
框架做：拿域写好的合并事件去渲染                         ← 机械活，框架
```

- **mergeKey 自动带域前缀** → 现阶段**天然只域内合并**，生日和信用卡揉不到一起。
- **合并总开关**（域级）+ **`mergeable`**（occasion 级，缺省 true，重要的人设 false）：
  重要的人 mergeKey 独一无二（或不填）→ 自动独立；普通的人共享 mergeKey → 合并。
  你要的"重要的人独立、其余合并"，一个字段搞定。
- **合并事件形态**（参考信用卡 `buildMergedTitle`，已验证的模式）：
  **类型标识进标题，明细进正文**——`🎂 今日3个生日` / 正文 `张三 60岁 / 李四 / 王五 55岁`。
  这正是 Ivan "域名称单独放标题、其余全在内容"的设想，与信用卡现有实现一致。
- **提前量不冲突**：合并只是日历清爽；提前量靠合并后那条事件挂多个 VALARM（[7,3,1] 天）解决。
- **信用卡现有合并可迁到此设施**：删掉它域内那套"按天分组"，改用框架分组 + 自己的
  mergeEvents 回调。代码变少，行为不变（实现时可选做，非强制）。

> **框架合并 vs 框架不碰业务，如何不矛盾**：框架做的是"分组"（机械、不懂业务），
> "合并成什么"是域写（`mergeEvents` 回调）。这与 ARCHITECTURE §0"框架永不做业务合并"
> **一致**——分组不是业务合并，写标题正文才是，后者仍在域。

#### (b) 强倾向（实现时的默认起点，Ivan 未最终锁定）

> **合并优先只作用于全天事件；定点事件倾向不参与。**

理由（Ivan 想到的正解）：全天事件只有"哪一天"、没有时刻，同天叠一起**不丢信息**；
定点事件带具体时刻（9:30 还款、14:00 签到），揉一条就**丢了时刻——到底几点？写哪个？**
这就是"时段不同会乱"的根源。所以**只有本来无时刻差异的全天事件，才谈得上无损合并**。

- 实现时**以"仅合并全天事件"为默认起点**；定点事件各自独立、时刻信息完整保留。
- 标为**强烈建议但未锁定**：Ivan 保留调整余地（如未来允许"同一时刻的定点事件"也合并，
  像信用卡 exact 模式那样——它合并的是同为 9:30 的还款，没丢时刻）。

#### (c) 留给未来（位置已留，难题标明，不写"永不"）

**跨域合并是已识别的未来方向**（Ivan 明确保留），但有未解难题，解决前不做：

- **未解难题一**：跨域合并的**混合正文谁写**？（框架不碰业务、单域不懂别域——
  "🎂生日+💳还款+✅签到"这条混合事件没有天然主人。）
- **未解难题二**：定点事件到底参不参与（见 (b)）。
- **架构已留位置**：合并是流水线**正式环节**，将来跨域是在**此环节内扩展**（识别跨域 mergeKey、
  解决混合正文归属），**不动事件对象结构、不动其他环节**。
- **框架级插件方向**：Ivan 提出"扩展框架能力"——合并环节本身将来可做成**可插拔**
  （如同域是插件、通道是插件）。**现在不设计其接口**（避免假预留），仅记为已识别的扩展方向。

> **为什么不留空的 `mergeGroup` 跨域字段**：那是假预留——字段留了，但混合正文难题一点没解，
> 通向一堵墙（参照被删的 needsFxPurchase）。**真预留是"合并环节可扩展"这个位置，不是空字段。**
> 位置留在流水线环节里，将来在原位填逻辑；空字段留在事件对象上，将来发现不够还得改结构。

### 6.2 长岁花名册 = 独立 kind，不参与合并

**一张年度花名册**，不是每人各发一条：

- `kind: 'roster'`，一年产**一条**（或按配置横跨两天），列**所有过对应历生日的人 + 岁数**。
  例：农历正月初一 → `🧧 今日全家长一岁：妈妈 61 / 爸爸 65 / …`（虚岁）。
- **可配置锚点**：农历新年（春节长岁，虚岁）、或公历某日（如 9/1，Ivan 提到的“大生日小生日”）。
- **可配置形态**：单条横跨两天（大年三十→初一，用 endDate）**或** 两天各一条（同内容不同 date）
  ——处理器内部 `return [事件A,事件B]` 的区别，不动结构（Ivan 已确认非骨架问题）。
- **不参与合并、不参与提前量**（它本身就是聚合视图）。
- **虚岁只对有年份的人算**（形态 A/B），形态 C/C′ 上不了花名册（§3 硬边界）。

## 7. 农历离线计算（库选型与锚点测试）

**农历是【不变的事实】**（1900–2100 表算完即钉死，无人发公告改闰月），与假期（会变，
归 workdays-core）**性质不同**——所以**不进 workdays-core**（详见 §8）。

### 7.1 库选型（2026-07 现状，实施时复核）

| 库 | 版本 | 体积 | 依赖 | 许可 | 判断 |
|---|---|---|---|---|---|
| **lunar-javascript**（6tail） | 1.8.x | ~512kB 未压 | 零 | MIT | **首选**：零依赖、活跃、生日够用 |
| tyme4ts（6tail 新作） | 1.5.x | ~728kB | 零 | MIT | 备选：设计更新，支持藏历（未来扩展时再看） |
| lunar-calendar / chinese-lunar | — | — | — | 停更 / **GPL** | 排除（GPL 不可用） |

- **体积不是问题**：Workers 限制是**压缩后** 3 MiB（免费）/ 10 MiB（付费）；512kB 中文串
  gzip 后约 120kB，随便放（见 AUTH 无关，纯部署事实）。

### 7.2 ⚠️ 铁律：只用转换，绝不用它自带的假期数据

**lunar-javascript / tyme4ts 都自带一套“法定假日/调休”数据**（tyme4ts v1.4.2 更新日志
明写“修复 2026-09-20 调休”）。**这与本项目铁律直接冲突**：

> 本项目**不持有任何假期数据**（假期事实唯一真相源 = workdays-core）。

**处置（复用信用卡 config 垫片 + 领地看守手法）**：

1. 全库**唯一**允许 import 农历库的文件：`src/domains/anniversary/lunar.js`，
   只导出转换函数（solar↔lunar），**绝不导出、绝不调用任何假期能力**。
2. **领地看守测试**（加进 render.test 框架层）：
   - `lunar-javascript` 只允许在 `domains/anniversary/lunar.js` 出现；
   - 任何地方出现 `HolidayUtil` / `getHoliday` / 该库假期 API = **红灯**。
   - 断言旁注明理由：**假期归 workdays-core 唯一真相源，此库自带的是竞品，不许碰。**

**用测试守边界，不靠自觉**（Ivan 验证过的手法）。

### 7.3 锚点测试（必须钉死，防库升级/API 猜错）

农历库的闰月 API 约定（负数表闰月？`isLeap` 标志？）实施时需**实测确认**，并钉一组已知答案
（放 render.test 零依赖层）：

- 2026/2027/2028 春节公历日期；
- 一个闰月年：闰月落在哪、普通月与闰月分别转出什么；
- **Ivan 家人的真实生日**（他手里有正确答案，最强验证）；
- 一个三十日 + 一个小月（验证 §4 三十→廿九政策）；
- 形态 A/B/C/C′ 各一例（验证析出边界）。

**钉死版本号（不用 `^`）**：库换版本、算法改了、API 猜错 → 测试立刻红。

## 8. 为什么不进 workdays-core（四条理由，翻案先读）

1. **性质不同**：农历是不变的事实（表算完钉死），假期是会变的事实（国务院年年发公告）。
   workdays-core 的整套机器（数据集插件/refresh 管线/归档/coverage 告警）是为“会变”造的，
   农历一样都不需要。
2. **身份不匹配**：它叫 workdays-core、描述是“多国假期/工作日事实”。塞农历 = 改身份，
   得是自觉决定，不能顺手。
3. **结构不匹配**：它按**国家码**组织数据集（CN/HK/SG）；农历没有国家码——跨 CN/HK/TW/VN/KR
   还各有变体。硬塞进国家码文件夹是拗。
4. **只有一个消费者**：为“以后可能有别人要”把农历塞上游，正是刚删掉的 needsFxPurchase
   “预留但从未启用”错误的跨仓库放大版。**过早提升很贵**（新包/新 CI/新版本线 + 搞浑上游身份，
   且改不回来），**将来真出现第二个消费者再提升很便宜**（一个纯函数 + 一张表）。

**结论**：农历转换住在域里（`src/domains/anniversary/lunar.js`），不动上游。

## 9. 日历标题（保持简单）

**标题 = 姓名 + 事件**，尽量短。详情进正文。

| 场景 | 标题 |
|---|---|
| 公历生日（非整寿） | `🎂 妈妈生日` 或 `🎂 妈妈 40岁生日`（岁数是否上标题：政策，缺省仅整寿上） |
| 农历生日 | `🎂 妈妈农历生日` / `🎂 妈妈农历40岁生日` |
| 整寿 | `🎂 妈妈 60岁生日`（自动升 high） |
| 纪念日 | `💍 结婚纪念日` / `💍 结婚10周年` |
| 长岁花名册 | `🧧 今日全家长一岁` |

- **虚岁/周岁全列进正文**（`虚岁 41 / 周岁 40`），闰月标注进正文/字段。
- **统一链接词、统一逻辑**（Ivan 定）：现在按“进日历”施工；将来进闹钟/todo 时，
  标题/reason/title 从同一套内容生成器出（同 CHANNELS §2 公共骨架），改起来快。

## 10. 时区

农历按**东八区**定义。境外使用时 `new Date()` 拿到当地日期，边界日农历可能错一天。
**处置**：本域事件**显式声明 `tz: 'Asia/Shanghai'`**（不吃框架缺省，同签到域“虚拟 UTC”/
EVENT-MODEL“墙上时间+tz”精神）。留 `?tz=` 参数覆盖（缺省东八），Ivan 提的“人在美国”场景
预留但当前默认东八。

## 11. 隐私（本域区别于信用卡的唯一新维度）

- **姓名只进 `config/anniversary.js` 和日历标题，永不进 URL、永不进 uid**（uid 用 person.id
  的哈希，不含姓名）。
- **本域 `private: true`**：所有通道输出（日历/闹钟/todo）都要 unlock（AUTH §3）。
  `?cal=all` 无 unlock 时不含本域（AUTH §4）。
- **诊断只打 id 不打姓名**（⚠️ 与信用卡区别对待）：信用卡诊断打 summary（卡名不敏感），
  但本域诊断若打 summary，会把**全家人姓名+年龄**列进默认开启、日历可见的诊断事件。
  故本域治理诊断**只打 uid（哈希）不打显示名**——`❌ anniv-a1b2c3-2026 越界`，
  不出现“妈妈 60岁”。**牺牲一点诊断可读性换隐私聚合保护。**
  （实现提示：governance 的 who() 对 private 域走“只打 uid”分支，或本域事件不填给诊断用的
  明文名。具体挂载点实施时定，原则是“private 域诊断不泄明文”。）
- **两条防线叠加**（分别防不同泄漏）：
  1. **分层防线**（AUTH §4.1）：没 unlock，本域根本不运行 → 诊断里连本域摘要都没有。
  2. **脱敏防线**（本条）：带 unlock 看得到本域诊断时，也只打哈希 uid 不打姓名。
  即使你自己带 unlock 看诊断，也不该在一屏里被列出全家姓名年龄——**聚合的姓名年龄比单条更敏感**。
- **默认走摘要档**（配合诊断分级，见开发计划“诊断分级重构”）：本域默认只报
  `🎂 纪念日:N 人 M 事件 ✅` 一行 + 异常；逐人明细只在 `?debug=verbose` 展开。
  这样即便本域挂了几十个人，默认诊断也不爆屏。

## 12. 骨架清单（四根主梁，施工时不得动摇）

这四条定错了要大改，是本规划书的地基：

1. **鉴权**：本域 private，两把钥匙分级 + 总开关，见 AUTH.md。
2. **两个正交维度**：kind（类型处理器）× calendar（历法转换器），各自可注册插槽，绝不焊死（§2）。
3. **合并**：`mergeKey` 域内分组设施（框架分组、域写内容、带域前缀防跨域）；全天优先（强倾向未锁定）；
   跨域合并留位置不留空字段（§6.1）。
4. **身份**：person 手写稳定 id + occasion 自动键（kind+calendar），uid 用 uidHash，删一条不牵连（§5）。

其余（政策缺省、字段名、标题格式、时区默认、库版本）全是**可在文档/config 改的肉**。

## 13. 交付物与实施顺序（建议）

**依赖顺序**：AUTH（鉴权）与 CHANNELS（todo 通道）是本域的前置——本域要 private 保护、
要能选 todo 通道。建议：

1. **先落 AUTH**（框架级，两个现有域受益，本域依赖）。
2. **再落 CHANNELS 的 todo 通道**（框架级，本域依赖）。
3. **最后落本域**：
   - 通讯录 config 模型 + 两层继承；
   - calendar 转换器插槽（solar/lunar）+ lunar.js 垫片 + 领地看守 + 锚点测试；
   - kind 处理器插槽（birthday/anniversary/roster）；
   - 政策参数（三级覆盖）；
   - 合并（复用信用卡路数）；
   - uid（uidHash + 自动键）；
   - 隐私（private 标记 + 诊断脱敏）。

**交付方式**：只交付新增/修改的单个文件，不打包。破坏性变更（本域全新，基本无）记 DEVLOG。
