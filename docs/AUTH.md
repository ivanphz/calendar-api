# 鉴权规划书 —— calendar-api 系统级 token（AUTH）

> **这份文档的地位**：框架级新增能力，独立于任何域、任何输出通道。它影响**所有**域和
> **所有** format（ics / json / todo / 未来的 bark…）。因此单独成篇，不塞进 EVENT-MODEL
> 或某个域文档。
>
> **状态**：逻辑已与 Ivan 逐轮敲定（模型自洽性已验证），**细节（参数名/头名/错误码文案）
> 可改，开发前需再完整过一遍**。冲突时以本文的【模型】为准，实现细节让路。
>
> **为什么现在做**：两个域时改鉴权是加法，五个域时改是手术。系统级收口要趁插件少。

---

## 0. 一句话模型

> **两种钥匙是【级别】关系，不是【并列】关系。**
> `unlock` 是含 `access` 权限的高级钥匙；不存在“只有 unlock 没有 access”的残缺态。

这一句话堵住了设计过程中发现的真漏洞（见 §11 决策记录）：若把两把钥匙设计成“并列两道门”，
总开关一关，access 那道门没了，private 的“双钥”就退化成“单钥”——自相矛盾。
改成“一把高级钥匙”后，private 的保护**与总开关完全无关**，任何时候进 private 都是同一把
unlock，不多不少。

---

## 1. 两种钥匙

| 钥匙 | 能开什么 | 谁持有 |
|---|---|---|
| **access**（普通钥匙） | 普通域（非 private）。**看不了 private** | 网关、想收紧后的普通订阅链接 |
| **unlock**（高级钥匙） | 普通域 **+** private 域。**天然含 access 全部权限** | 只有你自己看敏感域的那几条链接 |

**核心不变量**：`能看 private` ⟹ `能访问 API`。反之不成立。
因此校验逻辑里，**先判 unlock，再判 access**——持 unlock 即视为已通过 access。

---

## 2. 一个总开关

`config/hub.js` 里一个布尔：

```
requireAccess:
  false（默认）→ 普通域【裸链可看】，不需要任何钥匙  ← 你现在的行为，存量链接零改动
  true         → 普通域也要钥匙（access 或 unlock 均可）  ← 你想“全收紧”时一键切换
```

**总开关只管普通域这一道门要不要锁。它动不了 private 域**——private 永远只认 unlock。

### 完整状态表（背下这张表就懂全部）

| | 总开关关（默认 false） | 总开关开（true，全收紧） |
|---|---|---|
| **普通域**（签到 / 信用卡） | 谁都能看（裸链） | 要 access **或** unlock |
| **private 域**（生日 / 未来的日记…） | **只有 unlock 能看** | **只有 unlock 能看** |

- private 那一列**两种状态完全相同** —— 这就是“隐私层雷打不动”的正确形态。
- 普通域那一列随总开关变：关时裸链、开时要钥匙（unlock 因含 access 权限也能开普通域）。

**回答那个逼出正确模型的问题**：“总开关关的时候进 private，是不是只要 unlock？” ——
**是，就是 unlock，且这是唯一需要的那把**。不是“少了一把”，而是 unlock 本就是进 private 的
那把钥匙，自带访问权，与总开关无关。

---

## 3. private 是域的静态标记

`private` 是域对象上的一个静态属性（`registry.js` 注册的域声明里），**不是** URL 参数、
**不是**运行时判断：

```js
export const anniversaryDomain = {
  id: 'anniv',
  contract: 2,
  private: true,          // ← 本域数据敏感，永远只在 unlock 后可见
  // …
};
```

- 生日域标 `private: true`。
- 信用卡 / 签到**不标**（卡名只有后四位，不敏感；签到站点不敏感）。
- 将来加日记域 → 标 `private`，自动进 unlock 保护；加天气域 → 不标，随总开关。
- **纯加法，不改骨架。** 敏感与否是一次性声明，写在域上，框架统一执行。

## 4. `?cal=all` 的语义降级

`all` 不再恒等于“所有域”，它取决于**当前请求的钥匙级别**：

| 钥匙级别 | `?cal=all` 实际含义 |
|---|---|
| 无钥匙 / 仅 access | **all 减去所有 private 域**（= 普通域全集） |
| unlock | 真正的 all（含 private） |

**这是有意的语义变化，不是 bug**（写进 UPGRADE 文档提醒）。设计意图：
你可以把 `?cal=all` 这条链接分享给家人（他们没有 unlock），他们看到的是签到等普通域，
**看不到生日**；你自己带 unlock 的那条 `?cal=all` 才是完整的。

**显式点名 private 域但钥匙不够时**（如 `?cal=anniv` 但只有 access）：
不静默返回空，而是**明确拒绝**（见 §5），否则你会以为“今天没生日”而不是“我没权限看”。

### 4.1 诊断也分层（跟随 unlock，天然对齐）

诊断事件（默认开启的那条全天事件）**自动跟随钥匙级别**，无需额外机制：

- **没 unlock**：private 域压根不参与本次请求（被摘出 selectedIds），它的 `debugLines`
  **根本没产生** → 诊断里自然没有生日相关内容。这是摘除机制的**自然结果**，不是新功能。
- **带 unlock**：private 域参与，其 `debugLines` 正常汇入诊断。

⚠️ **但有一处必须显式堵**（不是自动的）：诊断里除各域 `debugLines`，还有**框架级汇总**
（“启用域: card + checkin + anniv”这类）。**没 unlock 时，框架级诊断也不得出现 private 域的
名字**——否则就泄漏了“存在一个生日域”，违反 §5“错误文案不列 private 域名”。
实现：框架级诊断在列举启用域时，无 unlock 则过滤掉 private 域名（只列可见域）。

## 5. 凭证提取：可扩展插槽（提取层 ≠ 判定层）

> 这是本鉴权设计里**最防返工的一条**。现在不分层，以后想加“header 传参 / 签名 / HMAC”
> 就是把散落在入口里的 URL 解析全刨出来重写——动全域的手术。分了层，加新方式只换一个函数。

**核心：钥匙【从哪来】与钥匙【够不够级别】是两件事，从第一天就劈开。**

```
请求 ──► [凭证提取器] ──► 中性结果 { level: none | access | unlock } ──► [判定层]
             ↑                                                              ↓
      未来在这里加新方式                                    §2 那张表、全部域、全部 format
      (URL / header / 签名 / HMAC / 时间戳…)                都只认这个中性结果
```

| 层 | 职责 | 稳定性 |
|---|---|---|
| **提取层**（credential extractor） | 从请求里把钥匙**弄出来**，翻译成级别。URL？header？签名？ | **可扩展插槽** —— 未来加新方式在此 |
| **判定层** | 拿到 `{level}` 后，按 §2 表决定可见域集合 | **稳定**，整个模型建在它上面，不动 |

这与本项目一以贯之的手法同源——**框架吃语法（怎么提取），判定吃语义（够不够级别）**；
也与 registry 的域插件、历法转换器表、通道后端适配器是**同一个“可注册插槽”模式**。

### 插槽契约

每个提取器实现同一接口：**输入请求，输出 `{ level, via }`（via 标记来源，用于诊断/日志），
提取不到就返回 `{ level: 'none' }` 让下一个提取器接手**。框架按顺序尝试注册的提取器，
**取到的最高级别生效**（一条请求同时带 URL access 和 header unlock，判定为 unlock）。

### v1 实现哪些、未来留什么（诚实边界）

| 提取器 | 状态 | 说明 |
|---|---|---|
| **URL 参数**（`?access=` / `?unlock=`） | **v1 实现** | iOS 订阅日历唯一可用（见 §7） |
| **HTTP header**（`Authorization: Bearer` 或自定义头） | **v1 实现** | 网关/脚本/手动调用优先用它，不进 URL 历史 |
| 签名 / HMAC / 时间戳防重放 | **预留插槽，不实现** | 未来更强鉴权照插槽注册第三个提取器，判定层与全部域零改动 |

> ⚠️ **预留的是“接口形状 + 一句照此加”，不是空实现。** 不写
> `extractHMAC(){ /* TODO */ }` 这种空壳——那是债（参照被删的 needsFxPurchase）。
> 留插槽（便宜、挡返工），不留空函数（贵、是债）。真要加签名时，照 URL/header 提取器的
> 样子写一个新的、注册进去即可。

## 6. 钥匙怎么传、错误怎么报

### 传输方式（两条都支持，各有用途）

| 方式 | 写法 | 适用 | 安全性 |
|---|---|---|---|
| **URL 查询参数** | `?access=xxx` / `?unlock=yyy` | **iOS 订阅日历唯一可用**（它不支持自定义 header） | 进浏览器历史/日志（见 §7） |
| **HTTP header** | `Authorization: Bearer xxx` 或自定义头 | 网关、脚本、手动调用 | 不进 URL 历史，稍强 |

**iOS 订阅日历只能用 URL 参数** —— 这是平台物理限制，不是偷懒。所以 URL token 是
“订阅日历这种场景能达到的安全上限”。能用 header 的调用方（如网关）优先用 header。

参数名 / 头名是**可改细节**，开发时定；本文只定“两种传输都要支持”。

### 错误响应（区分“没权限”和“钥匙错”）

| 情况 | 返回 | 说明 |
|---|---|---|
| 总开关开、普通域、无任何钥匙 | `401` | 需要 access |
| private 域、无 unlock（有没有 access 都一样） | `401`（或 `403`，实现时定） | 需要 unlock；**文案不得泄露“存在哪些 private 域”**——只说“需要更高权限”，不列域名 |
| 钥匙值不对 | `401` | 不区分“不存在”与“值错”，避免探测 |
| `?cal=all` 仅 access | `200` | 正常返回普通域全集（**不报错**，这是降级不是拒绝） |

**一条隐私铁律**：错误信息**绝不列出 private 域的存在**。“你无权访问 anniv 域”这种文案
等于告诉路人“这里有个生日域”。统一说“需要更高权限的钥匙”，不指名。

## 7. 安全边界（必须写死，别误判）

**token 是【能力钥匙】，不是【密码】。**

| | |
|---|---|
| **防得了** | 链接被随手转发、误分享给不该看的人（**你自用防手滑外泄的核心诉求**） |
| **防不了** | 能读你网关访问日志 / 浏览器历史 / iOS 账户设置的攻击者（URL 参数会落这些地方） |

- URL 里的 token 会进：浏览器历史、iOS「设置→日历→账户」、Cloudflare 访问日志、任何中间代理日志。
- 要防日志泄漏，只能用 header 传 —— 但 **iOS 订阅日历不支持 header**，所以订阅场景做不到。
- **能换即安全**：token 不是记在脑子里的密码，是可轮换的钥匙。疑似泄漏 → 改 config 里的值 →
  旧链接立即失效 → 重新订阅。所以它才**可以**进 URL。
- unlock 泄漏后果比 access 重（能看隐私），所以：**unlock 值应比 access 更长更随机，且更勤轮换**。

**别拿它当银行级鉴权**。它的定位是“自用系统防手滑外泄 + 给普通层加一道可选门”。
真要抗攻击者（而非防手滑），得上 OAuth 之类——那 iOS 订阅日历就用不了了，与本系统场景冲突。

## 8. 钥匙存哪（config 领地规矩）

钥匙值是**机密**，遵循现有领地规矩，但比普通 config 更严：

- **不写进仓库**。参照 `wrangler.toml` 现有做法：钥匙值放 **Cloudflare 环境变量 / Secret**
  （dashboard 配置，`keep_vars` 保护），**不进 `config/` 源码、不进 git**。
- `config/hub.js` 里只放**总开关** `requireAccess`（布尔，不敏感）和“**哪些钥匙名对应什么级别**”
  的结构，**钥匙的值从 `env` 读**。
- 框架校验时从 `ctx.env` 取钥匙真值比对（域拿不到、也不需要拿到钥匙——鉴权是框架入口的事，
  域只管自己 `private` 与否）。

> 这与信用卡 email 模型“收件地址放 Secret、不进代码”同一条铁律：**机密进 env，不进仓库。**

## 9. 改动范围（诚实清单）

**框架入口（worker-entry）**：新增一道鉴权前置——解析钥匙（URL + header）→ 定级别
（none / access / unlock）→ 按 §2 表决定可见域集合 → 把不可见域从 `selectedIds` 摘掉
（或整体 401）。**这是入口加一层，现有域逻辑零改动。**

**registry**：域声明支持 `private: true` 静态属性（不声明 = 普通域）。

**config/hub.js**：新增 `requireAccess` 总开关。

**存量链接**：
- 总开关默认 `false` → **你现在所有裸链接继续有效，零改动**。
- 哪天你把总开关拨 `true` → 那时才需要给普通域的存量链接（含网关那条）补 access。
  **改不改、何时改，你决定**，不是这次强制。
- 生日域从第一天就 `private`，其链接从一开始就带 unlock（新链接，无存量负担）。

**测试（render.test 框架层）**：
- 无钥匙 + 总开关关 → 普通域可见、private 不可见
- 无钥匙 + 总开关开 → 普通域 401
- access → 普通域可见、private 仍不可见
- unlock → 全可见（验证“高级钥匙含 access 权限”）
- `?cal=all` 各级别下的域集合正确
- 错误响应不泄露 private 域名（文案断言）
- 三个 format（ics/json/todo）鉴权行为一致（同一把锁，与格式无关）

## 10. 与三层生态的关系

鉴权是 **calendar-api 决策层自己的门禁**，与上下游解耦：

- **上游 workdays-core**：无关（它是 npm 包，进程内调用，无网络边界）。
- **下游 alarm-api**：网关持 **access**（普通钥匙）拉闹钟 json。它**拿不到 unlock**，
  所以**网关永远够不到 private 域的数据**——即使网关日志泄漏，漏的也只是 access + 普通域数据，
  生日安全。这正是“两种钥匙分级”最实在的收益。
- 敏感的是**数据**（谁的生日），不是**格式**——所以 ics / json / todo **同一把锁，无例外**。
  生日无论从哪个口出去都要 unlock。

## 11. 决策记录（为什么是这个形状，翻案前先读）

- **为什么两种钥匙而非一把**：一把钥匙既管访问又管隐私，就得放进网关配置（网关要拉闹钟），
  等于把“能看全家生日”的最高权限交给一个只需拉还款提醒的组件。网关日志一泄漏 = 最高权限泄漏。
  分成 access（给网关，够用）+ unlock（自留，看隐私），**分别泄漏、分别轮换**。
- **为什么是“级别”而非“并列两道门”**（本文核心，堵住的真漏洞）：并列模型下，总开关一关，
  access 那道门不设防，private 的“双钥”退化成“仅 unlock”——自相矛盾。改成
  “unlock 含 access 权限的高级钥匙”后，private 保护**与总开关无关**，模型自洽。
- **为什么保留总开关而非“access 一刀切所有请求”**：一刀切会强制**所有**裸链接（含最不敏感的
  签到）立即加 access，且丧失“普通层默认宽松”的灵活性。总开关默认 `false` 兼容全部存量裸链，
  想收紧时一键 `true`——**给你选择权，而非替你决定**。
- **为什么不做“域公开/私有 × 有无access × 有无unlock”的四象限**：过度设计，正是要避免的
  结构复杂化。收敛为“两把级别钥匙 + 一个总开关 + private 静态标记”，三件事各司其职，无冗余。
- **为什么 private 是静态标记而非 URL 参数**：敏感与否是域的**固有属性**，不该由调用方在
  URL 里临时声明（那样路人构造个参数就能改变可见性）。写在域声明上，框架统一执行，改不了。

## 12. 留给实施会话 / Ivan 现场确认

- 参数名与头名（`access`/`unlock`/`token`/`key`…；`Authorization: Bearer` 还是自定义头）。
- 错误码用 `401` 还是对 private 用 `403`（“已认证但无权”语义更准，但会暴露“存在受限资源”——
  倾向统一 `401` + 不指名文案，实施时定）。
- 是否需要**多把 access**（如给不同家人不同钥匙、可分别吊销）——当前设计是单把；若要多把，
  config 结构留一个 map 即可，不改模型。**默认单把，按需扩展。**
- unlock 是否需要**分域粒度**（未来多个 private 域，是否一把 unlock 通开全部 vs 每域一把）——
  当前设计是**一把 unlock 通开所有 private**。若未来隐私域增多需要分权，再引入“域→所需钥匙名”
  的映射；**现在不做**（YAGNI，避免重蹈 needsFxPurchase 覆辙）。
