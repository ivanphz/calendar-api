# 下游通道契约 —— CHANNELS（决策层 → 执行层的多通道出口）

> **这份文档的地位**：定义 calendar-api（决策层）向 alarm-api（执行层）输出的**通道**抽象。
> 现有 `?format=json` 闹钟是第一条通道；本次新增 `?format=todo`（提醒事项）是第二条；
> Bark / 邮件 / iOS 原生通知等是**预留的后续通道**。一份契约，N 个通道。
>
> **状态**：逻辑已与 Ivan 敲定，**细节（字段名/缺省值/format 名）可改，开发前再过一遍**。
> 与现状冲突时以仓库文档为准（ARCHITECTURE / EVENT-MODEL / DOWNSTREAM），冲突处**先停下问**。
>
> **鉴权**：所有通道出口都受 AUTH.md 约束（同一把锁，与 format 无关）。private 域的任何
> 通道输出都要 unlock。本文不重复鉴权细节，见 AUTH.md。

---

## 0. 一句话模型

> **通道对接是【框架】的事，不是【插件】的事。**
> 插件只说三件：**要不要进这个通道、提前多久、内容是什么**。
> 框架负责：把这三句话翻译成每个后端要的格式（json 闹钟 / todo / bark 推送 / 邮件…）。

这条是整个设计的地基。它带来两个直接后果：

1. **插件与通道后端解耦** → todo 通道和生日插件可**并行开发**（隔着一层薄契约）。
2. **加新通道 = 加一个后端适配器，契约不动** → 未来上 Bark/邮件，插件一个字不改。

## 1. 通道全景

| 通道 | format | 落地成 | 状态 |
|---|---|---|---|
| **alarm**（闹钟） | `?format=json` | iPhone 闹钟（穿透专注/静音） | ✅ 现有，协议 v1，见 DOWNSTREAM.md |
| **todo**（提醒事项） | `?format=todo` | iPhone 提醒事项（Reminders） | 🔨 **本次新增**，见 §3 |
| **bark** | （未定） | Bark 推送通知 | 🔲 预留，见 §5 |
| **mail** | （未定） | 邮件 | 🔲 预留，见 §5 |
| **native** | （未定） | iOS 原生本地通知 | 🔲 预留，见 §5 |

**为什么每个通道一个独立 format，而不是一个 format 里加 type 字段区分**：

> **独立 format 是 fail-safe。** 不认识 todo 的旧网关，请求 `?format=todo` 会得到它不认识的
> 结构（或它压根不会去请求这条 URL）——**它永远看不到 todo 数据**，杜绝“未知字段被静默
> 当闹钟处理”。若把 todo 混进 `format=json`，旧网关会拿到混合数据，把 todo 条目误当闹钟。
> 一条 format 一种消费者，各认各的，是**协议层的安全边界**。

## 2. 通道无关的公共契约（所有通道共享）

无论哪个通道，域交出的条目都遵循**同一套语义骨架**。差异只在“哪个 format 出口”和“个别
通道特有字段”。公共骨架：

| 字段 | 含义 | 归谁 |
|---|---|---|
| `uid` | 身份 + 配对锚点 | 域产出，**硬规则同闹钟侧**（见 §4） |
| `date` | 到期/触发日 `YYYY-MM-DD` | 域 |
| `time` | 到期/触发时刻 `HH:MM`（可选，缺省=全天，下游定缺省钟点） | 域 |
| `title` / `reason` | **人话**，给人看的（字段名各通道沿用习惯：闹钟叫 reason，todo 叫 title） | 域 |
| `severity` | **多重要**：`high` / `normal` / `low`（见 §2.1） | 域 |
| `tz` | IANA 时区名（可选，缺省东八），换算归**消费侧** | 域声明，网关换算 |

**平台中性判据**（每个字段过一遍，决定它能不能进公共契约）：

> *“这个字段换成 Android 的 Tasks / 换成邮件 / 换成 Bark，还成立吗？”*
> 成立 → 可进条目（date/time/title/severity/tz 都成立）。
> 不成立（iOS 的 list 名 / Urgent 标记 / Alert 音 / 归档语义）→ **那是下游源配置的领地，
> 不进条目**。域永远不知道 iPhone 概念（与闹钟通道“绝不输出 Gate-ES-* 标签”一脉相承）。

### 2.1 severity：域说“多重要”，通道落地不进条目

**severity 是平台中性的抽象**——它表达业务重要度，不表达“怎么落地”：

| severity | 语义（业务） | 各通道怎么落地（**下游源配置决定，不进条目**） |
|---|---|---|
| `high` | 重要，别错过 | 闹钟→响；todo→Urgent 列表/置顶；bark→高优先级推送；mail→标记重要 |
| `normal` | 常规 | 闹钟→响；todo→普通 Alert；bark→普通推送 |
| `low` | 知道就行 | todo→静躺不提醒；bark→静默；mail→普通 |

- **`critical` 不进本抽象**——那是闹钟通道的活（穿透专注/静音的最高级）。域要“吵醒你”就走
  alarm 通道，不是靠 severity 升级。severity 管的是“同一通道内怎么落地”，不是“换通道”。
- **severity → 具体落地（Urgent/Alert/静躺）的映射，写在下游源配置里，不在条目里**。
  域说 `high`，网关的 todo 源配置决定 `high` 映射到 iOS 的哪个列表/提醒级别。
  这就是“域说多重要，网关说怎么落地”的分界。

### 2.2 提前量归下游采纳窗口，域不管

**域算“哪一刻到期/触发”，落地节奏（提前几天躺进清单、提前几分钟推送）归执行层。**

- 与闹钟通道“采纳窗口≈2 天归网关”**同一分界**：业务算时刻，执行层定节奏。
- todo 的“提前 N 天出现在清单里”= 下游源级 `leadDays`（网关配置），**域不下发**。
- **例外**：日历本体（VALARM）的提前量**是域的事**（那是日历侧提醒，见 EVENT-MODEL §3）——
  别混淆。**通道的提前量归下游，日历的 VALARM 提前量归域。** 两套东西，两个归属。

## 3. todo 通道 v1（本次新增，施工目标）

### 3.1 适配器契约：新增可选第四出口

`build()` 返回值新增 `todos`，与 `events` / `alarms` / `debugLines` 并列：

```js
async build(state, hub, ctx) {
  return {
    events:     [ /* 日历本体 */ ],
    alarms:     [ /* 闹钟通道，协议 v1 */ ],
    todos:      [ /* 待办通道，v1，见 §3.2 —— 可选 */ ],   // ← 新增
    debugLines: [ /* 诊断 */ ]
  };
}
```

- **老域不返回 `todos` = 空数组**，非破坏性。**`contract` 版本号不动**（仍为 2）——
  这是可选出口的增补，不是契约破坏。
- 与 events/alarms 的关系：**同一业务事件允许多通道并存**。妈妈生日可以同时进日历（events）+
  提醒事项（todos）；信用卡还款可以同时进闹钟（alarms）+ 提醒事项（todos）。
  **各通道 uid 空间互不相识**（见 §4）。

### 3.2 todo 协议 v1 条目

```js
{
  uid: 'card-a1b2c3-202607',   // 必填。硬规则同闹钟侧（§4）：前缀=域id、[A-Za-z0-9_.-]、
                               //   ≤40、bucket 语义、【不含时钟时间】、跨请求稳定
  date: '2026-07-28',          // 必填。到期日
  time: '10:00',               // 可选。到期时刻；缺省=全天到期（下游自定缺省钟点）
  title: '💳 招行还款 ¥3,421', // 必填。人话，给人看
  notes: '账单月 2026-07 …',   // 可选。人话正文
  url: 'https://…',            // 可选。真实业务链接（身份标记由网关拼，不占用此字段）
  severity: 'normal',          // 必填。high | normal | low（critical 不进本通道——见 §2.1）
  tz: 'Asia/Shanghai'          // 可选。同闹钟协议，换算归消费侧
}
```

### 3.3 框架代劳（域零成本获得）

与闹钟通道对称，框架对 todos 做：

- **治理**：uid 双协议校验（复用 governance.js `checkAlarmUid` 路数）+ 全 feed 唯一性。
  违规 = **响亮熔断该条 + 诊断报数**，绝不静默修复（同闹钟侧哲学）。
- **按日未来过滤**：丢弃 `date <` 基准日（当日保留）。
- **排序**：按 date + time。
- **熔断域剔除**、**字段原样透传**（`tz` 等可选字段不增不删不改）。
- 出口壳：`{ v: 1, todos: [...] }`。

### 3.4 域侧配置：channel 选通道

信用卡域、签到域（及未来的生日域）在**条目/账户级**选进哪些通道：

```
channel: 'alarm' | 'todo' | 'both' | 'off'     // 命名与缺省值实施时与 Ivan 定
```

- `alarm` → 只进闹钟；`todo` → 只进提醒事项；`both` → 都进（两套 uid，见 §4）；`off` → 都不进（仅日历）。
- **缺省值分域定**：闹钟型业务（还款）可缺省 `alarm`；提醒事项型业务（生日祝福）可缺省 `todo`。
  由实施会话与 Ivan 现场定，写进各域 config 注释。
- **channel 是“进哪个通道”的开关；severity 是“进去之后多重要”**。两个正交，别混。

## 4. uid：跨通道的身份规则

### 4.1 硬规则（与闹钟通道完全相同）

todo uid 与闹钟 uid **同一套硬规则**（AUTH 无关，这是下游对账的物理要求）：
前缀=域id、字符集 `[A-Za-z0-9_.-]`、**≤40**、bucket 语义、**不含时钟时间**、跨请求稳定。
**治理全硬**：违规响亮熔断，复用 governance.js 的 `checkAlarmUid` 与 `uidHash`。

### 4.2 三套 uid 空间互不相识

一个业务事件若同时进日历、闹钟、todo，它有**三个 uid**，分属三个独立空间：

```
日历事件 uid：card-a1b2c3-202607-20260728   （EVENT-MODEL §4，可含日期）
闹钟   uid：card-a1b2c3-202607             （bucket，不含时钟时间）
todo   uid：card-a1b2c3-202607             （bucket，不含时钟时间）
```

- 闹钟和 todo 的 uid **可以长得一样**（都是账单月 bucket）——**不冲突**，因为它们在
  **不同 format 的输出里**，各自对账各自的下游存储（网关的闹钟表 vs 提醒事项表）。
  下游把它们当两个独立对象管理。
- **配对靠前缀**（同 EVENT-MODEL §4 那套）：三个 uid 共享 `card-a1b2c3-202607` 前缀 →
  肉眼看出是同一业务事件的三个通道投影。

### 4.3 改期 = 同 uid + 新 date/time（比闹钟侧更干净）

- **闹钟侧**：改时间靠“旧关新建”（uid 不含时间，网关拼标签时带时间）。
- **todo 侧更干净**：iOS 提醒事项**可原地修改**（Set Detail 实测可改 Due Date）——
  改期 = 同 uid + 新 date/time，下游 **upsert 自然生效，无需关旧建新**。
- 两者共享同一条铁律：**uid 不含时钟时间、跨请求稳定**。所以改提前量/假期漂移时 uid 不变。

## 5. 预留通道：Bark / 邮件 / 原生（形状预留，不实现）

**加一条通道 = 加一个后端适配器 + 一个 format，公共契约（§2）不动。**

未来加 Bark 的施工路径（示意，非本次实现）：

1. 新 format（如 `?format=bark`）——独立 format，fail-safe（§1）。
2. 复用 §2 公共骨架条目（date/time/title/severity/tz）——**域一个字不用改**，它早就在 build 里
   返回这些语义了；只是多一个出口去消费同一批条目。
3. 新增一个后端适配器：把公共条目翻译成 Bark 的 HTTP 推送格式；severity → Bark 优先级的映射
   写在**下游源配置**（不进条目）。
4. 域侧 `channel` 增加 `'bark'` 取值。

> ⚠️ **预留的是“这条路怎么走”的说明 + 公共契约的平台中性，不是空实现。**
> 不写 `formatBark(){ /* TODO */ }` 空壳（参照被删的 needsFxPurchase）。真要加时照 todo 的
> 样子建。**severity 从第一天就设计成平台中性**（§2.1），正是为了让未来加 Bark/邮件时
> 域零改动——这是**现在就要做对**的，不是预留。

**平台中性的验证**：§2 那句判据（“换成 Android/邮件/Bark 还成立吗”）就是为这些预留通道
把关的。任何进公共条目的字段都得过这关，否则未来某个通道会容不下它。

## 6. 下游对接登记（每个新增 todo 视图，一次性）

同 DOWNSTREAM.md §3 的路数，网关侧加一条源配置（下游领地）：

```js
{ name: '生日待办', type: 'todo', code: 'bday', url: 'https://<本Worker>/?cal=anniv&format=todo&unlock=…', enabled: true }
```

- **带 unlock 的 URL 是隐私 URL** → 走下游 **Cloudflare Secret**（`EXTERNAL_ALARMS` 或其
  todo 对应物），**不进明文 SOURCES**（同 AUTH §7 机密进 env 铁律 + DOWNSTREAM §3 隐私 URL 路数）。
- `code` 由下游分配，进提醒事项的身份标记；换视图 = 换一条源。
- todo 源的 `leadDays`（提前几天躺清单）、severity→列表映射，都在**这条源配置里**（下游领地）。

## 7. 联调与排错

- **本框架侧自检**：`?cal=<域>&format=todo&testDate=<某天>` 直接看条目 JSON。
  uid 稳定性/不抖（同 DEVGUIDE §10 第 3、4 条）是下游 todo 不抖的充分条件。
- **鉴权自检**：private 域的 todo URL 不带 unlock → 应被 AUTH 拒（见 AUTH §6 错误响应）；
  带 unlock → 正常出条目。
- **治理自检**：故意构造超长/含中文 uid → 应在诊断响亮熔断（同闹钟侧 K 组测试路数）。

## 8. 已裁决（连同理由，实施时不要翻案，翻案先问）

1. **uid 不含时间、不带版本号**。todo 可原地改（Set Detail 改 Due Date），改期=同 uid+新 date/time，
   下游 upsert 生效。与闹钟“bucket 锚事件不锚响铃”同构，且更干净（无需关旧建新）。
2. **severity 进条目，通道落地不进条目**。域说“多重要”，网关源配置说“怎么落地”（severity→
   Urgent/Alert/静躺 映射在网关）。
3. **提前量（提前几天躺清单）归下游采纳窗口**（网关源级 leadDays），域不管。与“闹钟采纳窗口
   ≈2 天归网关”同一分界。**注意别和日历 VALARM 提前量混**（那个归域）。
4. **治理全硬**（uid 前缀/字符集/长度/唯一性/无时钟时间），违规=响亮熔断该条，绝不静默修复。
   复用 governance.js `checkAlarmUid` 与 `uidHash`。
5. **独立 format=todo，不并入 format=json**。fail-safe：不认识 todo 的旧网关永远看不到它，
   杜绝“未知字段被静默当闹钟”。
6. **通道对接是框架的事，插件只说“进不进/提前多久/内容是什么”**。加新通道=加后端适配器，
   契约不动，域零改动。severity 从第一天平台中性。

## 10. 留给实施会话 / Ivan 现场裁决

- `channel` 配置的**粒度**（账户级够不够，还是要条目级）与**缺省值**（各域不同）。
- **签到域是否上 todo 通道**，还是先只做信用卡域试点。
- `?format=todo` 是否**复用视图参数全套**（tags/exclude/past/future）——倾向复用（一致性），
  确认无坑即可。
- 字段名微调（`title` vs `summary`、`notes` vs `description`，与 events 对齐还是与闹钟对齐）。
