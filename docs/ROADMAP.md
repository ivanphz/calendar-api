但它们本身对整个框架有价值），一个是域。

## 2. 实施顺序（推荐）

### 第一波：框架级能力（互不依赖，但有推荐次序）

**推荐 ②→③→④→① 或 ②→③→①→④**，理由：

1. **② AUTH 先做** —— 影响全局（所有 format 入口），且两个现有域立刻受益（能收紧访问了），
   生日域强依赖它。**越早做，后面越不用返工加鉴权。**
2. **③ CHANNELS todo 次之** —— 生日域强依赖，且是 ⑥ Bark 等的前置。
3. **④ 诊断分级** —— 趁现在只有 2 个域，重构成本最低；生日域进来会加一大段清单，
   现在定好分级结构，生日域从第一天按新结构产诊断（契约在生态长大前定死）。
4. **① 合并设施** —— 最独立，随时可插入；做完能让信用卡现有合并迁过去（代码变少）。

> 四个互不依赖，可并行或任意排。上面只是“如果串行做，这个顺序最省返工”。

### 第二波：生日域（前三波就绪后，且自身再拆四刀）

⑤ 生日域不要一次性全上，**分四刀**（每刀独立可测、可上线）：

1. **刀一 · 最小可用**：公历生日 + 进日历。验证通讯录模型、两层继承、uid 稳定编码。
   （不碰农历、不碰通道、不碰合并——先把骨架跑通。）
2. **刀二 · 农历**：接农历库（lunar.js 垫片 + 领地看守 + 锚点测试）。**这刀最易翻车**
   （库的闰月 API 要实测，用家人真实生日验证），单独一刀便于聚焦。
3. **刀三 · 类型扩展**：纪念日（anniversary kind）+ 长岁花名册（roster kind）。
   验证 kind 处理器插槽。
4. **刀四 · 通道接入**：把生日接进 todo / 闹钟通道（channel 字段）+ 合并（mergeKey）+
   隐私脱敏最终化。

> 每刀跑通、CI 绿、再下一刀。别攒着一次上——生日域是这批里唯一的信用卡级体量，
> 分刀是控制风险的关键。

### 第三波：通道扩展（按需，todo 稳定后）

⑥ Bark / 邮件 / 原生：**等 ③ 的通道抽象被 todo 实地验证过**（见 §4），
想要哪个加哪个，各半轮。

## 3. ④ 诊断分级重构（本文详述，无独立文档）

**问题**：现在的诊断是“把所有东西平铺列出”——每个域的账户/任务清单、治理段、coverage 段
全部展开。**这个模式随插件数线性爆炸**：2 个域够看，10 个域就是滚不完的墙。生日域还会
再加一大段人员清单。

**这不是“内容太多、删几行”能解决的，是结构问题。解法是分级，不是简化。**

> **“简化”会删掉排错命根**（那些告警是你查问题的依据，删哪条都有人依赖）。
> **“分级”内容不删，只是默认折叠**——与项目“响亮降级、绝不静默”一致：响亮是针对**异常**，
> 正常状态就该安静。现在诊断“正常状态也很吵”（列出每个账户），这才是要改的。

**分两级：**

| 档位 | 触发 | 内容 |
|---|---|---|
| **摘要档**（默认） | `?debug` 开（现状默认） | 每域**一行摘要**（`💳 信用卡:6账户 12事件 ✅`）+ **仅异常展开**（越界/熔断/coverage 缺口/废除参数告警）。正常状态安静——无告警就只有几行“✅ 某域 N 条”+ 窗口/口径基本信息 |
| **详细档** | `?debug=verbose` | 现在的全部内容：账户清单、任务清单、每条推算审计明细…… |

**设计要点：**
- **默认档随插件数增长仍可控**：10 个域 = 10 行摘要 + 若干告警，不爆屏。
- **异常永远露**（不论哪档）：越界裁剪、uid 熔断、coverage 缺口、废除参数、域构建失败哨兵
  —— 这些是“响亮”的对象，摘要档也必须显示。
- **明细按需展开**：`verbose` 才出的是“正常状态下的详情”（每个账户列出来这种）。
- **跟随 unlock**（AUTH §4.1）：没 unlock，private 域连摘要行都没有（它没运行）；
  框架级“启用域”列表也不列 private 域名。

**改动**：纯框架内部（worker-entry 组装诊断的那段）。**域不动**——各域照产 `debugLines`，
只是框架决定默认露多少、怎么分档。可能需要域把 `debugLines` 分成“摘要行”和“明细行”两类
（或框架约定 debugLines 第一行为摘要、其余为明细），具体接口实施时定。

**工作量**：小，1 轮。**时机**：趁 2 个域时做，生日域从一开始按新结构产诊断。

## 4. ⑥ Bark/邮件/原生：等什么才能开发？

**等的不是时间，是 ③ 的通道抽象“被 todo 实地验证过一次”。**

- CHANNELS 已为它们把图纸画好（公共骨架 date/time/title/severity/tz、severity 平台中性、
  独立 format 防误判）。
- **但图纸对不对，要 todo 真跑通才知道**。todo 是第一个“非闹钟”通道，它会把公共骨架、
  severity 落地映射、独立 format 这些设计**实地检验一遍**。
- **todo 跑通并稳定 = 通道抽象被证明成立 = Bark 照抄。** 那时加 Bark = 填一个后端适配器
  （把公共条目翻译成 Bark 的 HTTP 推送），severity→Bark 优先级的映射写在下游源配置，
  半轮的事。

**todo/bark/邮件/原生是同一个模式的四次复用**：todo 是第一次（最贵，顺带验证抽象），
后三次都便宜。**先让 todo 这个具体案例跑通，再谈扩展——别凭空造通用层。**

## 5. 贯穿所有工程的纪律（沿用 v6 已验证的做法）

1. **每个工程 = 给新 AI 的独立任务**：三份规划文档（AUTH/CHANNELS/PLUGIN-ANNIVERSARY）
   + 本 ROADMAP + **当前仓库**一起给。文档反复强调“以仓库现状为准，冲突先停下问”，
   别让新 AI 照文档凭空发挥。
2. **先推、先测、CI 绿了再下一个**：AUTH 和合并动了框架，回归风险高于纯加域——沿用 v6 的
   领地看守 + 域层/框架层双测试。
3. **契约在生态长大前定死**：④诊断分级、③通道契约，都要趁域少时定好，生日域从第一天遵守。
   这是 v6“eventLines→事件对象”那次的教训（等域多了再改契约，成本翻倍）。
4. **预留形状，不预留空实现**：合并环节可扩展（不留空 mergeGroup 字段）、通道后端可扩展
   （不留空 formatBark）、历法转换器可扩展、凭证提取器可扩展——**留位置，不留 TODO 空壳**
   （needsFxPurchase 的教训）。
5. **分刀上线**：尤其生日域，四刀各自可测可回滚，别攒大爆发。
